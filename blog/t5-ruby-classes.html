In a previous post I had introced the concepts of class while talking about Arrays and Hashes. What exactly is a class? Let's step back for a second and try to understand them from a high level.

There are two basic paradigms by which programs are written these days: via Object Oriented programming and via Procedural programming. Procedural programming defines programs as a series of checklist items that must be executed in-sequence, in order to yield an end result. Object Oriented programming, however, describes programs as a collection of building blocks that have specific behaviors and attributes that then interact with each other in differing ways. When said building blocks, termed Classes in Ruby, happen to communicate with other Classes, a set of behaviors will execute depending on the interactions of the attributes involved in the communication.

In Ruby, unlike in other Object Oriented languages, Objects consist of both behaviors as well as attributes. Other languages, like Java, take care to decouple these concepts into separate 'types' of elements. Because Ruby is a little unique in that every element is considered an Object, every single element requires its own defined set of behaviors before it can exist. In order to create an element we'll then need to call for one specific instance of the element that will then require a number of attributes to be passed in.

Let's walk through an example of a couple types of transportaion. Both cars and bikes are characterized as having certain default attributes: wheels, model, and year of production. Bikes not only differs from cars in the number of wheels they have, but they can differ from one another based on their individual models. We can create a basic object Class, called Transportation, that characterizes both bikes and cars in order to model this difference. In the process, we'll also ask the instances of our objects what they "are" by sending them a type message from which they will behave by reporting back information.

class Transportation
  attr_reader :wheels, :year, :model

  def initialize(year, model, wheels)
    @wheels = wheels
    @year = year
    @model = model
  end

  def type
    "This is a #{year} #{model}."
  end
  
end

------

bike = Transportation.new(2005, "Mountain Bike", 2)

p bike.type

=> "This is a 2005 Mountain Bike."

car = Transportation.new(2012, "Chevy Camaro", 4)

p car.type

=> "This is a 2012 Chevy Camaro."

p bike.wheels

=> 2

p car.wheels

=> 4

There's a lot going on here. First, we define a number of attributes that are required for the class upon initial class creation - which are year, model, and wheels (hence located in an initialize method). Depending on what we want to do with each new creation of Transportation (termed an instance), we've created behaviors that define what happens when the Ruby program interacts with each Transportation instance. Incidentally, these behaviors are what are known as methods, and that's what is between the def...end keywords. In this particular program we're doing nothing more than creating instances of transportaion, then asking the bike and car variables what they "are" and how many wheels they have by calling methods.

Classes are not really all that hard to understand. The hard part is trying to figure out how to create Classes that can both hold information as well as change over time as the scope of the program changes. We could just as easily have created seperate Car and Bike classes and gotten into much more detail if we really wanted to. This is, of course, where the creativity comes into play when deciding how best to engineer a solution to a problem.
